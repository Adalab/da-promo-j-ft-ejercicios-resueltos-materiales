Tipos de Ramas en Git 🌿

Cuando trabajamos con Git, hay varios tipos de ramas que suelen utilizarse comúnmente en un proyecto. Cada rama tiene un propósito específico y facilita la organización y el desarrollo del proyecto. Aquí te explico los tipos de ramas más comunes:

Rama main o master 🌟

Propósito: Esta es la rama principal de un repositorio. Por convención, master era el nombre predeterminado, pero recientemente muchos proyectos han cambiado a main para reflejar un lenguaje más inclusivo.

Uso: Contiene el código de producción, es decir, la versión estable del proyecto que está lista para ser desplegada o distribuida.

Ramas de desarrollo (develop) 🛠️

Propósito: Es una rama paralela a main y se utiliza para integrar las características y cambios que están en desarrollo.

Uso: Una vez que las nuevas características están completas y probadas, se fusionan en develop antes de ser finalmente integradas en main.

Ramas de características (feature) ✨

Propósito: Cada rama de características se utiliza para desarrollar una nueva funcionalidad o mejora específica del proyecto.

Uso: Estas ramas suelen crearse a partir de develop y, una vez que el desarrollo de la característica se completa, se fusionan de vuelta en develop.

Convención de nombres: feature/nueva-caracteristica.

Ramas de corrección de errores (bugfix) 🐛

Propósito: Similar a las ramas de características, pero se enfocan en corregir errores en el código.

Uso: Estas ramas se crean a partir de develop y, una vez que el error se corrige, la rama se fusiona de nuevo en develop.

Convención de nombres: bugfix/correccion-error.

Ramas de soporte (hotfix) 🔥

Propósito: Se utilizan para aplicar correcciones rápidas a la rama main sin pasar por develop. Son útiles para solucionar errores críticos que necesitan ser corregidos de inmediato.

Uso: Estas ramas se crean a partir de main y, una vez que la corrección está lista, se fusionan tanto en main como en develop para mantener la coherencia.

Convención de nombres: hotfix/correccion-critica.

Ramas de liberación (release) 🚀

Propósito: Preparar una nueva versión del proyecto para su lanzamiento.

Uso: Estas ramas se crean a partir de develop cuando se ha completado el desarrollo de una versión. En esta rama, se pueden realizar pruebas finales y correcciones de errores menores. Después, la rama se fusiona en main y develop.

Convención de nombres: release/v1.0.0.

🔄🔄🔄 Ejemplo de flujo de trabajo con ramas 🔄🔄🔄


Inicio del proyecto: Se crea el repositorio con la rama main.
Desarrollo inicial: Se crea una rama develop a partir de main.
Nuevas características: Se crean ramas de características (feature/nueva-caracteristica) a partir de develop. Una vez completadas, se fusionan en develop.
Corrección de errores: Se crean ramas de corrección de errores (bugfix/correccion-error) a partir de develop. Una vez corregidos los errores, se fusionan en develop.
Preparación para el lanzamiento: Se crea una rama de liberación (release/v1.0.0) a partir de develop. Después de las pruebas finales, se fusiona en main y develop.
Corrección rápida: Si se encuentra un error crítico en producción, se crea una rama de soporte (hotfix/correccion-critica) a partir de main. Una vez corregido, se fusiona en main y develop.


Este flujo de trabajo permite una organización clara y estructurada del desarrollo, facilitando la colaboración y el mantenimiento del proyecto.
